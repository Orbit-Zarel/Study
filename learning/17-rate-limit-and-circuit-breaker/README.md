# 17 限流与熔断基础

这一模块继续补你后面做平台型后端时必须有的治理能力：

- 限流
- 熔断

前一节你已经理解了 timeout、retry、fallback。

但真实系统里只靠这些还不够。

因为当上游持续变慢、持续报错、或者用户请求突然暴增时，你还需要系统具备“自我保护”能力。

## 1. 什么是限流

限流可以先理解成：

- 在单位时间内，只允许一定数量的请求通过

它最直接的目的就是：

- 防止系统被打爆
- 防止某个用户占满资源
- 防止把上游供应商配额打空

## 2. 为什么 LLM Gateway 一定会遇到限流

因为你的主项目后面会同时面对两类流量压力：

- 对外：业务方或用户不断调用你的接口
- 对内：你的网关不断调用 OpenAI 或通义千问

所以你至少会有两层限流意识：

- 用户级限流
- 供应商级限流

例如：

- 某个用户每分钟最多调用 20 次
- 某个 provider + model 每秒最多发 5 次请求

## 3. 限流最核心在控制什么

限流本质上控制的是：

- 请求通过速率

也就是：

- 不是所有请求都无限制立即执行

超过阈值时，系统要么拒绝，要么排队，要么降级。

当前阶段先建立最重要的直觉：

- 限流是系统保护机制，不是业务功能

## 4. 一个最小限流思路

你可以先把它理解成：

1. 系统接收请求
2. 根据某个 key 做计数，例如 `userId`
3. 检查当前时间窗口内是否超过阈值
4. 如果没超过，允许执行
5. 如果超过，直接拒绝

例如：

- `user-1001` 在 1 分钟内最多允许 3 次调用

第 4 次请求就应该被拒绝。

## 5. 什么是熔断

熔断可以先理解成：

- 当某个下游持续失败时，暂时不要再继续调用它

这和 retry 不一样。

retry 的思路是：

- 这次失败也许只是偶发，稍后再试一次

熔断的思路是：

- 这个下游现在大概率有持续性问题，再打下去只会放大故障

## 6. 熔断为什么重要

例如你的主 provider 是 OpenAI。

如果它连续超时、连续 5xx，而你的系统还在疯狂重试，就会出现：

- 线程被占住
- 请求大量堆积
- 整体延迟越来越高
- 连正常请求也被拖慢

这时更合理的动作通常是：

- 暂时打开熔断
- 直接快速失败
- 或直接走 fallback provider

## 7. 一个很基础的熔断直觉

你先记住三个状态就够：

### Closed

- 正常状态
- 允许请求继续访问下游

### Open

- 熔断打开
- 暂时不再访问下游

### Half-Open

- 过一段时间后，放少量请求试探下游是否恢复

## 8. 限流和熔断分别解决什么问题

### 限流解决

- 流量过大
- 资源被打满
- 配额被耗尽

### 熔断解决

- 下游持续异常
- 错误扩散
- 故障放大

## 9. 它们和前面几节怎么串起来

这几节其实已经形成一条完整链路：

- `15` 负责统一 LLM Gateway 抽象
- `16` 负责 timeout / retry / fallback
- `17` 负责 rate limit / circuit breaker

也就是开始从“能调起来”走到“更像线上系统”。

## 10. 当前阶段你应该先理解哪些点

先抓住这几个：

1. 限流是控制请求通过速率
2. 熔断是下游故障时的保护机制
3. retry 不是无脑做，熔断也不是一开始就开
4. 限流偏流量治理，熔断偏故障隔离

## 11. 学完这一模块后你应该能做到

- 理解用户级限流和供应商级限流的区别
- 看懂一个最小的时间窗口限流逻辑
- 理解熔断为什么常和 timeout、retry、fallback 一起出现
- 能把 rate limit / circuit breaker 放进你的 LLM Gateway 设计里
