# 26 MCP 与 A2A 进阶

这一模块开始补你当前实习和 Agent 岗都非常关键的一条线：

- `MCP`
- `A2A`

你前面已经接触过：

- MCP 工具开发
- 工具注册与挂载
- API Key 鉴权
- Agent 工程模式
- LangChain / LangGraph

这一节要做的，是把这些经验整理成更清晰的协议层与协作层认知。

## 1. 什么是 MCP

MCP 可以先理解成：

- 让模型或 Agent 以统一方式发现和调用外部工具的一套协议化接口

它解决的核心问题是：

- 工具怎么暴露
- 工具参数怎么描述
- 调用方怎么发现工具
- 工具结果怎么返回

所以 MCP 的重点不是某个单独工具，而是：

- 工具接入规范

## 2. 为什么 MCP 很重要

如果没有统一协议，Agent 调工具通常会变成：

- 每个工具一套私有格式
- 每个调用方一套适配逻辑
- 每换一个 Agent 框架就要重接一次

MCP 的价值就在于：

- 把工具接入标准化
- 把工具描述结构化
- 让模型能“知道有哪些工具可用”

## 3. MCP 里最核心的几部分

先抓住这几个：

1. tool schema
2. tool description
3. tool parameters
4. tool execution
5. tool result

也就是：

- 工具叫什么
- 是干什么的
- 要什么参数
- 怎么执行
- 返回什么

## 4. 什么是 tool schema

tool schema 可以先理解成：

- 工具的结构化说明书

它通常描述：

- name
- description
- input schema
- 参数类型
- 必填项

这一步非常重要，因为模型不是靠“猜”来调工具，而是靠 schema 理解工具。

## 5. 为什么参数描述很重要

很多工具调用效果差，不是模型太弱，而是工具描述太差。

例如这些问题都很常见：

- 参数名过于含糊
- docstring 太短
- 枚举范围没写清楚
- 时间范围含义不清楚

结果就是：

- 模型知道要调工具
- 但不知道该怎么填参数

所以一个好工具，重点不只是逻辑正确，还包括：

- 参数语义清楚
- description 可被模型理解

## 6. MCP Server 和 MCP Client 怎么区分

### MCP Server

可以先理解成：

- 暴露工具的一方

它负责：

- 注册工具
- 对外暴露调用入口
- 执行工具逻辑
- 返回结果

### MCP Client

可以先理解成：

- 调用工具的一方

它负责：

- 发现可用工具
- 选择工具
- 组织参数
- 发送调用请求

所以：

- Server 提供能力
- Client 消费能力

## 7. MCP 不只是“能调起来”，还包括权限和边界

真实项目里非常重要的一点是：

- 不是所有工具都该无条件暴露给所有调用方

常见控制包括：

- API Key 鉴权
- 用户权限过滤
- 工具白名单
- 参数级校验
- 输出脱敏

所以 MCP 的工程重点之一是：

- 工具暴露边界

## 8. 什么是工具发现与注册

在 MCP 体系里，一个关键问题是：

- 调用方怎么知道有哪些工具

通常这依赖：

- 工具注册
- 工具枚举
- 路由挂载
- schema 暴露

这也是为什么你前面做过的“自动扫描子包、挂载到某路径”很重要。

## 9. A2A 是什么

A2A 可以先理解成：

- Agent 和 Agent 之间的协作

它不是模型调工具，而是：

- 一个 Agent 把任务交给另一个 Agent
- 或多个 Agent 分工协作完成任务

所以：

- MCP 更偏 Agent 对工具
- A2A 更偏 Agent 对 Agent

## 10. A2A 和多 Agent 流水线有什么关系

多 Agent 流水线是更宽泛的工程模式。

A2A 可以理解成其中更明确的一类：

- Agent 之间有清晰的任务边界
- 会发生任务委派、结果回传、上下文传递

例如：

- Planner Agent 生成任务
- Research Agent 查资料
- DB Agent 查数据
- Writer Agent 汇总答案

这就已经很接近 A2A 思路了。

## 11. A2A 和 MCP 的本质区别

这两个概念很容易混。

### MCP

- 调的是工具
- 工具通常没有独立“智能体身份”
- 更偏能力调用

### A2A

- 调的是另一个 Agent
- 对方自己也可能会继续规划、调工具、整理结果
- 更偏任务协作

一句话理解：

- MCP 是“我调用某个能力”
- A2A 是“我把一个子任务交给另一个智能体”

## 12. A2A 里最关键的设计点

先抓这几个：

1. 任务边界
2. 上下文传递
3. 结果格式
4. 失败处理
5. 权限边界

如果这些没设计清楚，多 Agent 系统会很快变乱。

## 13. 什么是上下文传递

在 A2A 里，一个 Agent 通常不会把全部历史都原样交给另一个 Agent。

更常见的是传：

- 任务目标
- 必要背景
- 限制条件
- 期望输出格式

这样做的原因是：

- 控制上下文大小
- 避免信息污染
- 让子 Agent 只关注自己职责

## 14. 什么是结果汇总

多 Agent 协作后，通常还需要一层汇总。

例如：

- 一个 Agent 查数据库
- 一个 Agent 查文档
- 一个 Agent 做分析

最后需要：

- 统一整理成最终回答

这也是为什么前面学的 `Synthesizer` 角色仍然重要。

## 15. 什么场景适合 MCP

适合：

- 数据库查询工具
- 文档读取工具
- 搜索工具
- 报表工具
- 日期计算工具

这些能力更像：

- 明确输入
- 明确输出
- 不需要独立智能决策

## 16. 什么场景适合 A2A

适合：

- 复杂任务拆解
- 多领域分工
- 多数据源协同分析
- 子任务之间角色差异明显

这些场景里，单 Agent 容易上下文膨胀，A2A 更自然。

## 17. MCP / A2A 在企业 Agent 里的常见组合

很多真实系统不是二选一，而是：

1. 主 Agent 接收用户问题
2. 主 Agent 通过 A2A 分配子任务
3. 各子 Agent 再通过 MCP 调用工具
4. 最终由汇总 Agent 生成回答

这也是更完整的平台型 Agent 思路。

## 18. 当前阶段最该先掌握什么

先抓住这几个：

1. MCP 是工具接入协议
2. A2A 是 Agent 间协作模式
3. tool schema 和参数描述直接影响调用质量
4. 工具暴露必须考虑权限与边界
5. A2A 的核心不是“多”，而是“任务边界清楚”

## 19. 学完这一模块后你应该能做到

- 能区分 MCP 和 A2A 的定位
- 理解 MCP server / client 的角色分工
- 理解 tool schema 为什么重要
- 理解多 Agent 协作时为什么必须控制上下文和结果格式
- 能把你已有的 MCP 实战经验放进更完整的 Agent 平台视角里
