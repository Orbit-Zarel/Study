# 24 向量库与混合检索

这一模块承接上一节 `RAG`。

上一节你已经知道：

- 文档要切块
- 文本可以做 embedding
- 检索时会取 top-k

这一节继续补：

- 向量库是什么
- `Faiss` 和 `Milvus` 在解决什么问题
- 什么是稠密检索
- 什么是稀疏检索
- 什么是 hybrid search

## 1. 为什么 RAG 后面一定会遇到向量库

因为只要你开始做 embedding 检索，就会遇到一个实际问题：

- 大量文档块的向量存在哪里
- 检索时怎么快速找相似向量

如果只有十几条数据，内存里遍历也能做。

但文档一多，就需要更专业的检索存储层。

这就是向量库出现的原因。

## 2. 什么是向量库

向量库可以先理解成：

- 专门用来存文本向量，并支持相似度检索的系统

它通常负责：

- 存储 embedding
- 建索引
- 相似度检索
- top-k 返回
- metadata 过滤

所以它不是普通数据库的简单替代。

它更像：

- 为语义检索优化过的数据层

## 3. 什么是稠密检索

稠密检索就是：

- 先把问题和文档都变成 embedding
- 再根据向量相似度找最相关内容

它擅长：

- 语义相似
- 同义表达匹配
- 非字面一致的召回

例如：

- “收入增长”
- “营收提升”

这种不完全同词的表达，稠密检索往往也能召回。

## 4. 什么是稀疏检索

稀疏检索更接近传统信息检索。

你可以先理解成：

- 根据关键词、词频、倒排索引来找文档

最常见的代表就是：

- `BM25`

它擅长：

- 精确关键词匹配
- 专有名词匹配
- 编号、代码、缩写匹配

例如：

- 股票代码
- 文件编号
- 医学术语
- 法条编号

这类内容，稀疏检索往往比纯 embedding 更稳。

## 5. 为什么只用稠密检索不够

因为真实问题里，经常同时存在两类需求：

- 语义相似
- 关键词精确命中

如果只用 embedding，可能出现：

- 语义接近，但缺少关键术语
- 专有名词召回不稳定
- 编号和实体名误召回

所以很多生产系统不会只做单一路径检索。

## 6. 什么是 hybrid search

hybrid search 可以先理解成：

- 把稠密检索和稀疏检索结合起来

也就是：

- 一部分靠 embedding
- 一部分靠关键词
- 最后再合并结果

这样做的目标是：

- 同时拿到语义能力和精确匹配能力

## 7. 一个最小 hybrid search 直觉

你可以先记成：

```text
用户问题
  -> dense retrieval
  -> sparse retrieval
  -> merge results
  -> top-k
```

后面更复杂时，还会再加：

- rerank

## 8. 什么是 metadata filter

很多时候你不是想“在所有文档里搜”。

而是想：

- 只搜某个公司
- 只搜某个年份
- 只搜某个报告类型
- 只搜某个用户有权限看到的内容

这时就要加：

- metadata filter

例如：

- `company = CATL`
- `year = 2025`
- `report_type = earnings_call`

这一步对企业系统非常关键。

## 9. `Faiss` 和 `Milvus` 各是什么

当前阶段先这样理解：

### `Faiss`

- 更偏向量检索库
- 常用于本地实验、小型服务、原型验证
- 适合先把语义检索跑起来

### `Milvus`

- 更偏完整向量数据库
- 更适合更正式的服务化场景
- 支持更完整的集合管理、索引、过滤、扩展能力

你当前学习阶段先抓住这个层次区别就够。

## 10. 向量库不是“存文本”的重点，而是“检索文本向量”的重点

很多初学者会误解成：

- 文档放进去就结束了

其实重点不在“存”，而在：

- 怎么索引
- 怎么召回
- 怎么过滤
- 怎么合并多路结果

所以真正影响效果的，往往不是“用了哪个库”本身，而是：

- chunk 策略
- embedding 模型
- top-k
- filter
- hybrid 方案

## 11. 一个最小生产化检索链路通常是什么样

先记住这条链路：

1. 文档切块
2. 生成 embedding
3. 写入向量库
4. 保存 metadata
5. 用户提问
6. 生成 query embedding
7. 相似度检索
8. metadata 过滤
9. 合并或排序结果
10. 返回给 RAG 上层

## 12. 当前阶段最容易混淆的几个点

### 向量库不等于 RAG

向量库只是 RAG 的一层基础设施。

### embedding 不等于最终答案

embedding 只负责帮助召回相关内容。

### hybrid search 不等于 rerank

hybrid 是多路召回。

rerank 是对召回结果再排序。

这两步不是一回事。

## 13. 你当前阶段最该先掌握什么

先抓住这几个：

1. 稠密检索看语义
2. 稀疏检索看关键词
3. hybrid search 是把两者结合
4. metadata filter 决定检索边界
5. 向量库是 RAG 检索层的重要基础设施

## 14. 学完这一模块后你应该能做到

- 理解为什么 RAG 后面通常要接向量库
- 能区分稠密检索、稀疏检索、混合检索
- 理解 `Faiss` 和 `Milvus` 的大致定位
- 理解 metadata filter 在企业系统里的价值
- 为下一节 `rerank` 和评测做准备
