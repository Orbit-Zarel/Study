# 14 Idempotency 与 RequestId

这一模块会直接衔接你刚学过的任务状态机。

因为一旦你开始做“提交任务”接口，就会马上遇到一个非常现实的问题：

- 同一个请求如果被发了两次，会发生什么

例如：

- 用户不小心连续点了两次“提交任务”
- 前端因为网络抖动自动重试
- 代理层或客户端重复发送请求

如果系统没有保护，就可能出现：

- 创建两条一模一样的任务
- 同一个动作被执行两次
- 重复调用大模型
- 重复扣费

这就是幂等要解决的问题。

## 1. 什么是幂等

你可以先把“幂等”理解成一句非常朴素的话：

- 同一个操作重复执行多次，结果应该和执行一次一致

注意，这不是说：

- 系统完全没有重复请求

而是说：

- 即使重复请求来了，也不要产生重复副作用

## 2. 为什么幂等在任务系统里特别重要

因为你的项目里有很多“不能重复”的动作：

- 创建任务
- 发送 Kafka 消息
- 调用大模型
- 写任务结果

这些动作如果被执行两次，问题会非常大。

所以任务系统里经常要重点保护：

- 任务提交接口
- 消费者执行逻辑

## 3. 什么是 RequestId

`requestId` 可以先理解成：

- 一次业务请求的唯一标识

例如：

- 前端在发起“创建任务”请求时，带一个唯一的 `requestId`
- 后端拿到后，用它判断“这个请求是不是已经处理过了”

## 4. 一个最基础的幂等思路

你可以先记住下面这条最常见的逻辑：

1. 客户端发起 `POST /api/tasks`
2. 请求里带上 `requestId`
3. 服务端先检查这个 `requestId` 是否已经存在
4. 如果不存在，就正常创建任务
5. 如果已经存在，就不要再创建第二条任务
6. 直接返回第一次创建出来的结果，或者返回“已处理”的结果

## 5. 为什么 `requestId` 很适合任务提交接口

因为任务提交属于“有副作用”的操作。

比如：

- 创建数据库记录
- 推送 Kafka 消息

这类操作最怕重复。

所以一个很经典的设计就是：

- `POST /api/tasks` 支持 `requestId`

这样即使用户点两次提交按钮，只要 `requestId` 一样，后端就知道：

- 这是同一件事，不应该重复创建任务

## 6. 幂等不是所有接口都一样需要

这点也很重要。

例如：

### `GET /api/tasks/1`

查询接口本身通常天然更接近幂等。

因为你查 1 次和查 10 次，通常不会产生额外副作用。

### `POST /api/tasks`

创建接口就不一样。

如果没有幂等保护，调用两次就可能创建两条任务。

所以你当前阶段先重点记住：

- 有副作用的写操作，更需要考虑幂等

## 7. 数据库里通常怎么体现幂等

一种非常常见的方式是：

- 在任务表里增加 `request_id` 字段
- 给这个字段加唯一约束

例如：

```text
tasks
├── id
├── request_id   (unique)
├── user_id
├── status
└── created_at
```

这样做的意义是：

- 即使代码漏掉了一层判断
- 数据库也会帮你挡住重复写入

所以幂等通常不只是“代码里判断一下”，还会配合数据库约束。

## 8. 幂等和状态机有什么关系

它们是强相关的。

例如：

- 任务第一次提交时，创建状态为 `PENDING`
- 如果同一个 `requestId` 又来一次
- 系统不应该再创建第二个 `PENDING` 任务

否则你会得到两条逻辑上相同、状态却独立变化的任务，这会让系统非常混乱。

所以可以这么理解：

- 状态机是在约束“一个任务怎么流转”
- 幂等是在约束“同一个任务不要被创建两次”

## 9. 幂等和 Kafka 有什么关系

这也很关键。

因为如果任务被重复创建，往往也会带来：

- 重复发送 Kafka 消息
- 重复消费
- 重复执行任务

所以任务提交流程里，幂等通常是 Kafka 前面的一道重要保护。

## 10. 当前阶段先怎么理解“返回旧结果”

当服务端发现：

- 这个 `requestId` 之前已经处理过

常见做法包括：

### 做法 A

直接返回第一次创建出来的任务信息。

### 做法 B

返回一个提示，告诉客户端这是重复请求。

在很多业务场景里，做法 A 会更实用，因为：

- 对客户端来说体验更顺
- 它只需要知道“这次任务已经存在了”

## 11. 一个最小任务提交幂等流程

你可以先把它记成下面这样：

1. 客户端生成 `requestId`
2. 调用 `POST /api/tasks`
3. 后端根据 `requestId` 查数据库
4. 如果没查到，就创建任务
5. 如果查到了，就直接返回已有任务

这个流程会是你后面项目里非常高频的设计点。

## 12. 学完这一模块后你应该能做到

- 理解幂等的基本含义
- 理解为什么任务提交接口需要 `requestId`
- 理解为什么数据库唯一约束很重要
- 能把幂等和状态机、Kafka 关联起来看
