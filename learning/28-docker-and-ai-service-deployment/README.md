# 28 Docker 与 AI 服务部署基础

这一模块开始补平台型 Agent 系统落地时绕不开的一层：

- `Docker`
- AI 服务部署

前面你已经补了：

- Agent 工程模式
- LangChain / LangGraph
- RAG
- MCP / A2A
- Prompt 与 model routing

这一节要补的是：

- 为什么本地跑通不等于可部署
- Docker 在 AI 服务里解决什么问题
- 一个 AI 服务部署时通常有哪些组成部分

## 1. 为什么需要 Docker

很多项目在本地能跑，在别的机器就会出问题。

常见原因：

- Python 版本不一致
- 依赖版本不一致
- 系统库缺失
- Node / Java / 浏览器依赖不一致
- 环境变量缺失

Docker 的核心价值就是：

- 把运行环境一起打包

也就是：

- 不只是代码可复制
- 环境也可复制

## 2. 什么是 Docker

当前阶段先把 Docker 理解成：

- 用镜像描述运行环境
- 用容器实际运行服务

你可以粗略理解成：

- `Dockerfile` 负责定义“这台小环境怎么做出来”
- `container` 负责“按这个环境真正跑起来”

## 3. AI 服务为什么尤其需要 Docker

因为 AI 服务经常依赖很多环境细节：

- Python 包
- 模型 SDK
- 系统级依赖
- 浏览器或 OCR 依赖
- 向量库客户端
- Node 工具链

这类项目如果只靠“手工装环境”，后面会很难维护。

## 4. 一个最小 AI 服务通常有哪些部分

先记住这几个：

1. 应用代码
2. 依赖安装
3. 环境变量
4. 启动命令
5. 健康检查

如果再复杂一点，还会有：

- Redis
- PostgreSQL / MongoDB
- 向量库
- MCP Server
- Nginx / 网关

## 5. 什么是 Dockerfile

Dockerfile 可以先理解成：

- 构建镜像的说明书

它通常会写：

- 基础镜像
- 安装依赖
- 复制代码
- 设置工作目录
- 配置启动命令

例如一个 Python 服务最常见的几步就是：

1. 选一个 Python 基础镜像
2. 安装 requirements
3. 复制代码
4. 启动 FastAPI / 服务进程

## 6. 什么是基础镜像

基础镜像就是：

- 你构建环境时拿来打底的镜像

例如：

- `python:3.12-slim`

这决定了：

- 运行时语言版本
- 底层系统环境

选基础镜像时通常要平衡：

- 体积
- 兼容性
- 依赖安装难度

## 7. 什么是环境变量

AI 服务部署时，很多配置不应该写死在代码里。

最常见的就是通过环境变量注入：

- API Key
- 数据库连接串
- 模型地址
- Redis 地址
- 日志级别

所以部署时通常要明确区分：

- 代码逻辑
- 运行配置

## 8. 什么是启动命令

容器最终还是要跑一个主进程。

例如：

- `uvicorn app.main:app --host 0.0.0.0 --port 8000`

这一步决定容器启动后真正跑什么服务。

对 FastAPI 项目来说，这通常就是：

- Web 服务入口

## 9. 什么是健康检查

健康检查可以先理解成：

- 用来判断服务现在是不是正常活着

常见方式：

- HTTP `/health`
- HTTP `/ready`
- 返回简单状态字段

这很重要，因为部署后你需要知道：

- 服务是否起来了
- 依赖是否连通
- 是否能接请求

## 10. 本地跑通和可部署的区别

很多人会误以为：

- 本地 `python main.py` 能跑
- 就等于完成了部署准备

其实差很多。

一个真正可部署的服务，通常还要考虑：

- 环境隔离
- 依赖锁定
- 启动方式标准化
- 健康检查
- 配置注入
- 日志输出

## 11. Agent / RAG 服务部署时常见的额外依赖

这类服务经常不只是一个 Python 进程。

还可能依赖：

- Redis
- PostgreSQL
- MongoDB
- 向量库
- MCP 服务
- 文件存储
- 浏览器运行时

所以真实部署时很常见的情况是：

- 一个主服务
- 多个配套服务

## 12. Docker Compose 可以怎么理解

当前阶段可以先把它理解成：

- 用一个配置把多个本地开发容器一起拉起来

例如：

- app
- redis
- postgres

这样你在本地就能模拟更接近真实系统的运行方式。

## 13. AI 服务部署时最常见的问题

高频问题通常是这些：

1. 环境变量没配
2. 依赖漏装
3. 容器里端口没暴露
4. 服务启动了，但依赖没连上
5. 本地路径在容器里不存在
6. 模型 Key 或数据库账号没有正确注入

## 14. 为什么日志和部署是连在一起的

部署后排障时，第一手证据通常不是代码，而是日志。

所以服务部署时至少要做到：

- 日志打到标准输出
- 关键错误能在容器日志里看到
- traceId / requestId 最好可追踪

这和你前面学的可观测性是连起来的。

## 15. 一个最小部署闭环该怎么理解

你可以先记成：

```text
代码
  -> Dockerfile
  -> 构建镜像
  -> 启动容器
  -> 注入环境变量
  -> 暴露端口
  -> 健康检查
  -> 接收请求
```

## 16. 当前阶段最该先掌握什么

先抓住这几个：

1. Docker 解决的是环境可复制问题
2. Dockerfile 决定镜像怎么构建
3. 容器启动后必须有明确主进程
4. 配置要通过环境变量注入
5. 健康检查是部署后验证服务状态的重要手段

## 17. 学完这一模块后你应该能做到

- 理解 Docker 在 AI 服务里的作用
- 理解 Dockerfile、镜像、容器之间的关系
- 理解一个最小 FastAPI / Agent 服务部署链路
- 理解为什么健康检查、环境变量、日志输出是部署基础
- 为后面更正式的服务化和云部署打基础
