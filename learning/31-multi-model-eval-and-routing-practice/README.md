# 31 多模型评测与路由实战

这一模块承接前面的 model routing。

前面你已经知道：

- 不同任务不一定该走同一个模型
- 失败时可能需要 fallback

这一节继续补：

- 模型路由策略到底怎么制定
- 多模型怎么比较
- 评测结果怎么反过来影响路由

## 1. 为什么需要多模型评测

因为“感觉这个模型不错”不等于它就适合你的任务。

不同模型可能在这些维度上差异很大：

- 成本
- 速度
- 结构化输出稳定性
- 工具调用能力
- 推理质量
- 引用准确率

## 2. 常见评测维度

先抓这些：

- latency
- cost
- answer quality
- tool use success rate
- JSON format success rate
- citation quality

## 3. 为什么评测和 routing 是连在一起的

因为路由策略不该拍脑袋定。

更合理的是：

- 先做评测
- 再根据任务类型决定主模型和备模型

## 4. 一个常见 routing 思路

例如：

- 简单问答 -> 小模型
- 文档总结 -> 大上下文模型
- 工具调用 -> tool-use 更稳的模型
- 结构化输出 -> JSON 更稳的模型

## 5. 什么是任务维度评测

不是所有模型都要用同一组题简单跑一遍。

更合理的是按任务分桶：

- 简单 QA
- RAG 问答
- 工具调用
- 结构化抽取
- 深度分析

这样得出的结论才更能指导 routing。

## 6. 评测结果通常怎么用

常见做法：

- 为每类任务选主模型
- 为关键任务选备用模型
- 为低价值请求选低成本模型

## 7. 当前阶段最该先掌握什么

先抓住这几个：

1. 多模型评测不是只看总分
2. 要按任务类型分别比较
3. routing 策略应该由评测支撑
4. 主模型与备模型都要提前设计

## 8. 学完这一模块后你应该能做到

- 理解多模型评测和 routing 的关系
- 理解常见评测维度
- 能按任务类型思考模型选择
- 为后续做模型对比实验打基础
