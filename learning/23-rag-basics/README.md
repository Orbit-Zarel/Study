# 23 RAG 基础

这一模块开始补 Agent 岗里最常见的一条主线：

- `RAG`

你前面已经补了：

- `LangChain` 组件层
- `LangGraph` 工作流层
- Agent 工程模式

这一节要做的，就是把“检索增强生成”这件事先建立最小闭环直觉。

## 1. 什么是 RAG

RAG 可以先理解成：

- 先检索外部知识
- 再把检索结果交给模型生成答案

也就是：

- Retrieval
- Augmented
- Generation

它解决的是：

- 模型参数里没有最新知识
- 业务知识不在模型训练数据里
- 需要基于私有文档回答问题

## 2. 为什么 Agent 不能只靠模型记忆

因为很多真实问题都依赖外部数据。

例如：

- 公司内部知识库
- 最新财报
- 医疗文献
- 平台数据库
- 用户上传文档

这些内容：

- 可能模型没见过
- 可能已经过时
- 可能只能在你的私有系统里拿到

所以 Agent 要想回答得更准，往往需要 RAG。

## 3. 一个最小 RAG 链路长什么样

你先记住这条链路：

```text
用户问题
  -> 检索相关文档片段
  -> 把片段拼进 prompt
  -> 模型基于这些片段回答
```

这就是最小 RAG 闭环。

## 4. 什么是文档切块

文档切块可以先理解成：

- 不把整篇文档直接塞给模型
- 而是先拆成较小片段

例如一篇长文档会拆成：

- chunk 1
- chunk 2
- chunk 3

这样做的原因很简单：

- 整篇文档太长
- 检索粒度太粗
- 不利于找到最相关内容

## 5. 为什么需要 chunking

chunking 的核心价值是：

- 提高检索精度
- 降低上下文浪费

如果一篇 50 页文档只作为一个整体去检索，会出现：

- 相关片段被淹没
- 不相关内容过多

拆块以后，系统更容易只取到真正有用的那几段。

## 6. 什么是 embedding

embedding 可以先理解成：

- 把文本变成向量表示

这样系统就能做：

- 语义相似度检索

也就是用户问题和文档块，不用完全字面一致，也能匹配上。

例如：

- 问题里写“收入增长”
- 文档里写“营收提升”

embedding 检索仍然可能把它们匹配到一起。

## 7. 什么是 top-k 检索

top-k 可以先理解成：

- 从所有文档块里取最相关的前 K 个

例如：

- top-3
- top-5

这一步解决的是：

- 不可能把所有文档块都塞给模型
- 只能先选最相关的一小部分

## 8. 什么是 augmentation

augmentation 可以先理解成：

- 把检索结果拼回 prompt 上下文

例如：

```text
问题：
请总结公司本季度利润变化。

参考资料：
1. ...
2. ...
3. ...
```

模型不是凭空回答，而是基于“参考资料”回答。

## 9. RAG 和普通聊天的区别

普通聊天通常是：

- 用户问题 -> 模型直接回答

RAG 则是：

- 用户问题 -> 先查知识 -> 再回答

所以 RAG 的关键不只是“模型会不会答”，而是：

- 检索有没有取对
- 取回来的内容够不够
- 拼接方式是否合理

## 10. 一个最小 RAG 系统通常有哪些组件

先记住这几个：

1. 文档源
2. chunking
3. embedding
4. 检索器
5. prompt 模板
6. 模型生成

这其实正好和上一节的 `LangChain` 组件层对应起来。

## 11. RAG 最容易出问题的地方

当前阶段先记住 4 个高频问题：

1. chunk 切得太大
2. chunk 切得太碎
3. top-k 取太少
4. 取回了相关文档，但 prompt 没把上下文组织好

所以 RAG 不只是“接一个向量库”。

真正效果好不好，取决于整条链路。

## 12. 什么时候适合用 RAG

适合：

- 私有知识问答
- 文档问答
- 报告总结
- 基于资料生成回答

不太适合：

- 完全不依赖外部知识的简单问题
- 只靠模型常识就能回答的内容

## 13. 为什么已经有联网搜索，RAG 还会被大量使用

这是一个非常重要的问题。

因为很多人刚接触 Agent 或大模型应用时，都会自然想到：

- 既然模型可以联网搜索
- 为什么还要专门做 RAG

核心原因是：

- 联网搜索和 RAG 解决的不是同一类问题

### 联网搜索更像什么

联网搜索更像：

- 去公网查开放信息

例如：

- 新闻
- 官网资料
- 最新公开公告
- 网络文章

### RAG 更像什么

RAG 更像：

- 从你指定的知识源里检索内容，再让模型基于这些内容回答

例如：

- 公司内部知识库
- 私有文档
- 业务数据库
- 用户上传文件
- 受控报告集合

## 14. RAG 出现并被广泛应用的原因

### 1. 很多知识根本不在公网

很多真实业务依赖的内容是：

- 内部文档
- 私有数据库
- 权限受限资料
- 用户上传附件

这些内容联网搜索通常拿不到。

但 RAG 可以把这些内容接进来。

### 2. 你需要可控的数据范围

联网搜索的问题是：

- 搜到哪些站点不完全可控
- 内容质量不稳定
- 版本不稳定
- 可能有转载、过时、噪声信息

而 RAG 的检索范围通常由你自己定义：

- 只查某个知识库
- 只查某个数据表
- 只查某些文档集合
- 只查指定时间范围的数据

### 3. 你需要可追溯的答案来源

很多业务场景里，答案不能只“看起来合理”。

还要能回答：

- 这段结论来自哪份文档
- 来自哪条记录
- 来自哪一页或哪一段

RAG 更容易做到：

- 返回命中的 chunk
- 返回引用片段
- 返回文档来源和记录 ID

### 4. 你需要和业务数据结构绑定

真实系统里常常不是“搜网页”，而是：

- 查财报片段
- 查 PostgreSQL 记录
- 查 MongoDB 文档
- 查带 metadata 的知识块

RAG 可以把这些能力组合起来：

- 语义检索
- 元数据过滤
- 权限控制
- 业务字段约束

### 5. 你需要稳定、可评测、可优化

联网搜索更适合开放问题。

企业应用更在意：

- 检索范围稳定
- 相同问题结果波动小
- 可以调 chunk 策略
- 可以调 top-k
- 可以加 rerank
- 可以做评测和持续优化

RAG 更适合这种系统化迭代。

### 6. 你需要安全与合规

很多行业场景不能把问题直接交给公网搜索：

- 金融
- 医疗
- 法务
- 企业内部分析

RAG 更适合在受控环境中接私有知识。

## 15. 如果模型已经有 web search，RAG 的优势是什么

RAG 的优势主要是这些：

- 私有知识可接入
- 检索范围可控
- 答案来源可追溯
- 更容易做权限管理
- 更容易做评测和优化
- 更容易和结构化业务数据结合

所以不要把问题理解成：

- 有 web search 了，还要不要 RAG

更准确的理解是：

- web search 适合开放知识
- RAG 适合私有知识和受控知识

## 16. 在 Agent 系统里，web search 和 RAG 是什么关系

在 Agent 系统里：

- `web search` 更像一种工具
- `RAG` 更像一种知识接入架构

一个 Agent 完全可以同时拥有这两种能力：

1. 先查内部 RAG
2. 如果内部知识不够，再走 web search
3. 最后统一汇总回答

这也是很多企业级 Agent 更常见的做法。

## 17. 如何理解 chunk

`chunk` 可以直接理解成：

- 从原始文档里切出来的一小段可检索内容

RAG 系统通常不会拿整篇文档直接做检索。

更常见的是：

- 先把文档拆成很多片段
- 每个片段作为一个检索单元

用户提问时，系统检索的其实不是整篇文档，而是这些 chunk。

## 18. 为什么必须切 chunk

因为整篇文档直接检索通常会有几个问题：

- 检索粒度太粗
- 相关片段容易被无关内容淹没
- 塞给模型的上下文浪费太多

所以 chunk 的核心价值是：

- 把知识切成更细的检索单元
- 把喂给模型的上下文控制在合理范围

## 19. chunk 大小如何描述

所谓 chunk 大小，本质上就是：

- 一个检索单元里包含多少文本

常见描述方式有几种：

### 按字符数

例如：

- 500 个字符
- 1000 个字符

### 按 token 数

这通常更常见，也更贴近模型实际处理方式。

例如：

- 200 tokens
- 400 tokens
- 800 tokens

### 按句子或段落数

例如：

- 3 段
- 5 句

### 按文档结构切

例如按这些边界切：

- 标题
- 小节
- 段落
- 表格说明

对财报、论文、制度文档，这种方式通常比纯长度切块更合理。

## 20. chunk 太大或太小会发生什么

### chunk 太大

- 一个 chunk 里混了太多主题
- 检索命中后噪声太多
- 真正有用的信息被稀释

### chunk 太小

- 一段完整语义被切碎
- 证据不完整
- 检索到了也可能回答不完整

所以合理 chunk 的标准不是数字本身，而是：

- 这个片段是否能独立表达一个局部意思

## 21. 什么是 overlap

很多系统切 chunk 时还会加：

- overlap

也就是：

- 相邻 chunk 之间保留一部分重复内容

目的通常是：

- 避免关键句正好被切断
- 保持上下文连续性

## 22. chunk 的常见切分策略有哪些

常见可以分成 5 类：

### 1. 固定长度切分

最简单的做法。

例如：

- 每 500 个字符切一块
- 每 300 个 tokens 切一块

实现方式通常是：

- 从头到尾顺序扫描文本
- 到达阈值就切一刀
- 可选加 overlap

优点：

- 简单
- 容易快速跑通

缺点：

- 容易把句子或段落切断

### 2. 句子级切分

做法是：

- 先分句
- 再按若干句组成一个 chunk

实现方式通常是：

- 先做句子分割
- 逐句累加
- 到达大小阈值后输出 chunk

优点是比固定长度更自然。

### 3. 段落级切分

做法是：

- 以段落为基本单元
- 一个或多个段落组成一个 chunk

实现方式通常是：

- 先按空行、段落标签或文本结构切成段落
- 再按长度阈值合并相邻段落

它通常更适合报告、论文、长说明文。

### 4. 结构化切分

做法是：

- 按标题、小节、列表、表格说明、章节边界切

实现方式通常是：

- 先解析文档结构
- 以结构节点为基础切块
- 如果某个结构块太大，再做二次细分

对财报、制度文档、技术文档，这种方式通常更合理。

### 5. 语义切分

做法是：

- 不只看长度
- 而是看主题是否发生变化

实现方式通常是：

- 先切成句子或段落
- 再比较相邻单元的语义相似度
- 语义明显断开时切分

## 23. 工程里最常见的组合策略

真实项目里通常不是只用一种切法。

更常见的是组合方案，例如：

- 结构化切分 + token 限制
- 段落切分 + overlap
- 句子切分 + 语义合并
- 标题切分后，对超长块再做长度切分

当前阶段最实用的理解是：

- 很多工程系统最终常用的是结构化切分 + token 限制 + overlap

## 24. 如何按语义切分

按语义切分，本质上就是：

- 不按固定长度硬切
- 而是在主题发生变化的地方切

目标是让一个 chunk 尽量只包含一个局部主题。

## 25. 按语义切分的典型实现思路

最常见的是这几种：

### 1. 相邻句子或段落相似度切分

做法：

1. 先把文档切成句子或段落
2. 给每个单元计算 embedding
3. 比较相邻单元的相似度
4. 如果相似度明显下降，就在附近切分

可以理解成：

- 前后两段还在讲同一件事，就继续合并
- 前后两段突然不像了，就切开

### 2. 滑动窗口比较语义变化

做法：

- 不只看前一句和后一句
- 而是看前一小段和后一小段的整体语义差异

例如：

- 左窗口是句子 1 到 3
- 右窗口是句子 4 到 6

如果左右窗口相似度明显下降，就认为中间可能是一个语义分界点。

### 3. 结构切分后再做语义细分

这是工程里更常见的实用方案。

做法：

1. 先按标题、段落、章节做第一层切分
2. 对超长结构块再在内部做语义切分

好处是：

- 比纯长度切分更自然
- 比全文纯语义切分更稳定

## 26. 按语义切分时通常会做哪些约束

即使已经做了语义判断，工程里通常还会加这些约束：

- 最小 chunk 长度
- 最大 chunk 长度
- token 上下限
- overlap

原因是：

- 纯语义切分可能会过碎
- 或者某一块过大

所以实际系统里，语义切分通常不会裸跑。

## 27. 语义切分适合什么场景

更适合：

- 长报告
- 财报
- 学术论文
- 法律和制度文档
- 主题切换明显的长文本

不太值得一开始就上语义切分的场景通常是：

- FAQ
- 很短的说明文
- 结构本来就很规整的小文档

## 28. 如何理解 token

`token` 可以理解成：

- 模型处理文本时使用的基本计数单位

它不是：

- 字数
- 单词数
- 字符数

而是模型把文本切分后得到的一段段最小处理片段。

## 23. 为什么 token 很重要

模型很多关键约束都直接和 token 有关：

- 上下文窗口大小
- 输入长度限制
- 输出长度限制
- 调用成本
- 响应速度

在 RAG 里更重要，因为你要同时装下：

- system prompt
- 用户问题
- 检索结果
- 模型输出

所以很多 RAG 参数，本质上都在管理 token 预算。

## 29. token 和 chunk 的关系

在 RAG 场景里，很多时候最关心的是：

- 一个 chunk 大概有多少 token
- top-k 召回后总共会占多少 token
- 最终上下文会不会超限

所以实际工程里，chunk 大小往往最终会回到：

- 按 token 控制

## 30. 如何提升 RAG 的准确率

提升 RAG 准确率，核心是看整条链路，而不是只盯最终回答。

通常优先看这几层：

### 1. 提升召回质量

先检查：

- chunk 是否合理
- top-k 是否过小
- embedding 模型是否适合当前领域
- 是否缺少 metadata filter
- 是否需要 hybrid search

### 2. 优化 chunk 策略

高频做法：

- 调整 chunk 大小
- 加 overlap
- 按文档结构切块

### 3. 做混合检索

很多场景只用 dense retrieval 不够稳。

尤其是这些内容：

- 实体名
- 股票代码
- 文件编号
- 缩写词

这时常见做法是：

- dense + BM25

### 4. 加 rerank

常见思路：

- retrieval 先取 top-10 或 top-20
- rerank 后只保留前 3 或前 5

它的作用是：

- 把真正最有帮助的证据排到前面

### 5. 优化 prompt 和上下文拼装

例如：

- 明确要求模型只基于提供资料回答
- 没证据时明确说不知道
- 按相关性排序后再拼装 context
- 减少无关 chunk 噪声

### 6. 建立评测集

不要只靠主观感觉调系统。

至少要准备：

- 问题
- 参考答案
- 应命中的证据
- 不应命中的干扰内容

然后再看检索和回答是否真的变好。

## 31. RAG 准确率排查顺序

实际排查时，可以按这个顺序看：

1. 正确证据是否被切进 chunk
2. 正确 chunk 是否被召回
3. 召回后是否排进前几名
4. 拼给模型的上下文是否被噪声污染
5. 模型回答是否忠于上下文

这个顺序通常比直接换模型更有效。

## 32. 你当前阶段最该先掌握什么

先抓住这几个：

1. RAG 是“先查再答”
2. chunking 决定检索粒度
3. embedding 支持语义检索
4. top-k 决定取多少上下文
5. augmentation 决定怎么把资料喂给模型
6. chunk 大小最终通常要按 token 来理解
7. 不同文档适合不同 chunk 切分策略
8. RAG 准确率问题要按链路逐层排查

## 33. 学完这一模块后你应该能做到

- 理解最小 RAG 闭环
- 理解 chunking、embedding、top-k 的作用
- 理解为什么联网搜索不能替代 RAG
- 理解 chunk、overlap、token 在 RAG 里的位置
- 理解 chunk 切分策略与语义切分的基本实现
- 理解如何系统性提升 RAG 准确率
- 能区分“模型回答问题”和“RAG 回答问题”的差异
- 为下一节向量库与混合检索做准备
