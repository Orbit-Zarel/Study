# 13 Task Status Machine

这一模块是你后面主项目里非常关键的一部分。

因为一旦你开始做“异步任务中心”，就一定会遇到一个问题：

- 一个任务现在到底处于什么状态

例如：

- 刚创建
- 已入队
- 正在执行
- 执行成功
- 执行失败
- 被取消

如果没有一套明确的状态规则，项目很快就会变乱。

## 1. 什么是状态机

你可以先把状态机理解成：

- 一个对象在生命周期里会经历若干个状态
- 这些状态之间不是随便乱跳的
- 只能按照允许的规则进行变化

例如任务：

```text
PENDING -> QUEUED -> RUNNING -> SUCCEEDED
```

这就是一种最基础的状态流转。

## 2. 为什么任务需要状态机

因为异步任务和同步接口不一样。

同步接口通常是：

- 请求来了
- 立刻做完
- 马上返回结果

而异步任务通常是：

- 先提交
- 稍后执行
- 执行中可能失败
- 失败后可能重试
- 用户可能在中途查询状态

所以如果没有状态机，你会很难回答：

- 这个任务现在到底走到哪一步了
- 这个任务是否还能重试
- 这个任务能不能取消

## 3. 一个最小任务状态集合

你当前阶段先记最常见的一版就够：

- `PENDING`
- `QUEUED`
- `RUNNING`
- `SUCCEEDED`
- `FAILED`
- `CANCELED`

### `PENDING`

表示：

- 任务刚创建
- 还没真正进入执行队列

### `QUEUED`

表示：

- 任务已经进入消息队列
- 等待消费者处理

### `RUNNING`

表示：

- 消费者已经开始真正执行任务

### `SUCCEEDED`

表示：

- 任务执行成功完成

### `FAILED`

表示：

- 任务执行失败

### `CANCELED`

表示：

- 任务被取消，不再继续执行

## 4. 为什么状态不能乱跳

这是状态机最重要的直觉之一。

例如下面这些跳转通常是合理的：

- `PENDING -> QUEUED`
- `QUEUED -> RUNNING`
- `RUNNING -> SUCCEEDED`
- `RUNNING -> FAILED`
- `PENDING -> CANCELED`

但下面这些通常就不合理：

- `SUCCEEDED -> RUNNING`
- `FAILED -> PENDING`
- `CANCELED -> RUNNING`

如果状态随便改，就会出现：

- 一个已经完成的任务又被当成未完成
- 一个被取消的任务又偷偷执行
- 前后逻辑互相矛盾

## 5. 为什么状态机对查询接口很重要

因为用户调用：

- `GET /api/tasks/{id}`

本质上最关心的就是：

- 当前状态是什么
- 成功了没有
- 失败了没有
- 是否还在处理中

所以状态不仅是系统内部逻辑，也直接影响用户体验。

## 6. 状态机和 Kafka 的关系

这两者在你的主项目里会强相关。

一种典型流程是：

1. 创建任务时，状态是 `PENDING`
2. 成功发送 Kafka 消息后，状态改成 `QUEUED`
3. Consumer 开始执行时，状态改成 `RUNNING`
4. 执行完成后，状态改成 `SUCCEEDED` 或 `FAILED`

所以你可以把 Kafka 理解成：

- 帮任务从“等待执行”进入“真正执行”的桥梁

## 7. 为什么状态机和幂等也有关系

如果消息重复消费了，而你没有做好状态保护，可能出现：

- 一个已经 `SUCCEEDED` 的任务又被重新执行
- 一个已经 `FAILED` 的任务被重复写结果

所以在实际项目里，经常会结合状态判断来做保护，例如：

- 如果任务已经是终态，就不再重复执行

## 8. 什么是终态

终态可以先理解成：

- 到了这个状态后，任务生命周期基本结束

常见终态：

- `SUCCEEDED`
- `FAILED`
- `CANCELED`

一旦进入终态，通常就不应该再随意改回处理中状态。

## 9. 状态机为什么能让代码更稳

因为它帮你建立了一套明确规则：

- 哪些状态允许转移
- 哪些状态不允许转移

好处是：

- 逻辑更清晰
- 更容易排错
- 更容易避免脏状态
- 更容易做重试和取消

## 10. 当前阶段你可以先怎么理解“重试”

你现在先不用把自动重试做复杂。

可以先这样理解：

- 如果任务失败了，可能允许用户手动触发重试

例如：

- `FAILED -> QUEUED`

但这个动作也应该是被明确定义的，而不是随便在代码里乱改状态。

## 11. 学完这一模块后你应该能做到

- 理解为什么异步任务一定要有状态
- 知道状态机不是“多此一举”，而是在约束系统行为
- 看懂最基本的任务状态流转
- 理解终态、重试、取消这些概念为什么都和状态相关
