# 状态机在项目里到底怎么体现

这份文档专门回答一个非常关键的问题：

- 状态机在项目里到底是一个什么东西
- 它是不是只是代码里的一小段判断
- 它和数据库、接口、Consumer、重试有什么关系

如果你第一次接触“状态机”这个词，觉得它有点抽象，这是很正常的。

其实在后端项目里，状态机通常不是某一个单独文件，而是“贯穿多个层次的一套规则”。

## 1. 先给出一句最短答案

在项目里，状态机通常会同时体现在下面几个地方：

1. 数据库里的状态字段
2. Java 代码里的状态定义
3. Service / Consumer 的状态流转校验
4. 接口行为的限制规则
5. 前端或调用方看到的任务生命周期

也就是说：

- 状态机不是只存在于某一个类里
- 它是一套贯穿系统的状态约束机制

## 2. 第一层体现：数据库字段

最直接的一层，就是数据库里会有一个状态字段。

例如任务表：

```text
tasks
├── id
├── user_id
├── type
├── status
├── created_at
└── updated_at
```


这里的 **status** 就是在保存：

* 这个任务当前到底处于什么阶段

例如：

* **PENDING**
* **QUEUED**
* **RUNNING**
* **SUCCEEDED**
* **FAILED**
* **CANCELED**

所以当用户调用：

* **GET /api/tasks/{id}**

本质上经常就是在读取这个状态字段。

## 3. 第二层体现：Java 里的状态定义

在代码里，通常不会到处直接写字符串，而是会定义一个枚举。

例如：

`<span><span>public</span><span></span><span>enum</span><span></span><span>TaskStatus</span><span> {     PENDING,     QUEUED,     RUNNING,     SUCCEEDED,     FAILED,     CANCELED } </span></span>`

为什么要这样做？

因为这样更清晰，也更安全：

* 少写错字符串
* 更容易统一管理状态
* 更容易做流转判断

所以你可以把 **enum TaskStatus** 理解成：

* “代码层面对状态集合的正式定义”

## 4. 第三层体现：Service 里的状态流转规则

这通常是状态机最核心的落地点。

例如：

* 创建任务时：**PENDING**
* 成功投递 Kafka 后：**QUEUED**
* Consumer 开始处理时：**RUNNING**
* 执行成功后：**SUCCEEDED**
* 执行失败后：**FAILED**

这些规则通常不会随便散落，而是会在 Service 里被集中控制。

例如你可能会有这种方法：

public void updateStatus(Long taskId, TaskStatus expectedCurrent, TaskStatus next) {
    // 先检查当前状态是不是 expectedCurrent
    // 再决定是否允许更新
}

也可能会有：

public boolean canTransition(TaskStatus current, TaskStatus next) {
    // 只允许规定的状态变化
}

也就是说，状态机不只是“存一下状态”，而是：

* 要控制状态怎样从一个阶段进入另一个阶段

## 5. 第四层体现：Consumer 里的执行保护

在你的项目里，这一点尤其重要。

因为 Kafka Consumer 收到消息时，不能直接盲目执行。

它通常会先做这些事：

1. 查任务当前状态
2. 确认是不是 **QUEUED**
3. 如果是，才能进入 **RUNNING**
4. 如果任务已经是终态，就不要重复执行

例如：

* 任务已经 **SUCCEEDED**
* 但因为消息重复投递，Consumer 又收到一次

这时如果没有状态保护，就可能重复调用 LLM、重复写结果、重复扣费。

所以状态机也会直接参与“消费幂等保护”。

## 6. 第五层体现：接口行为限制

状态机会直接影响接口允许做什么。

例如：

### /cancel

如果当前状态是：

* **PENDING**
* **QUEUED**

可能允许取消。

但如果当前状态是：

* **SUCCEEDED**
* **FAILED**

通常就不允许再取消。

### /retry

如果当前状态是：

* **FAILED**

可能允许重试。

如果当前状态是：

* **RUNNING**

通常不应该让用户再重试。

所以状态机会直接影响：

* 某个接口能不能调
* 调了以后应该返回成功还是错误

## 7. 第六层体现：前端和用户视角

状态机不是只有后端自己关心。

用户或前端同样会依赖这些状态：

* **PENDING**：任务刚提交
* **QUEUED**：等待处理
* **RUNNING**：正在执行
* **SUCCEEDED**：可以查看结果
* **FAILED**：可以提示失败原因
* **CANCELED**：显示任务已取消

所以状态机也决定了：

* 页面如何显示
* 按钮是否可点
* 用户应该看到什么提示

## 8. 状态机不是“只写一个 if 判断”就结束

很多人一开始会以为状态机就是：

* 写几个 **if-else**

这当然是最基础的一部分，但在项目里它通常会扩散到很多地方：

* 数据库存什么状态
* 代码里允许哪些转换
* Consumer 是否继续执行
* 接口是否允许取消或重试
* 前端显示什么按钮和文案

所以更准确地说：

* **if-else** 只是状态机规则的一种代码表达方式
* 但状态机本身是整套任务生命周期管理规则

## 9. 一个最小例子：任务提交流程

在你的项目里，一次任务提交的状态机落地大概会是这样：

### 数据库

* 插入任务，状态 = **PENDING**

### Service

* 保存任务后发送 Kafka 消息
* 发送成功后把状态改成 **QUEUED**

### Consumer

* 消费消息时先检查状态是不是 **QUEUED**
* 如果是，更新成 **RUNNING**
* 执行完后更新成 **SUCCEEDED** 或 **FAILED**

### 查询接口

* 返回当前状态给调用方

### 重试接口

* 只允许 **FAILED -> QUEUED**

这就是一个完整的状态机在系统里的体现。

## 10. 为什么你会联想到 Airflow

这个联想非常合理。

因为 Airflow 这类调度系统，本质上也非常依赖任务状态。

例如它也会有：

* **queued**
* **running**
* **success**
* **failed**

这说明你已经开始抓到“状态机是任务系统核心思想”这件事了。

区别在于：

* Airflow 是完整的工作流调度平台
* 你现在学的是一个更轻量的、项目内部的任务状态机实现

但底层思想是相通的：

* 任务不能乱跳状态
* 状态决定系统下一步能做什么

## 11. 你当前阶段最值得记住的结论

请先记住这几句：

1. 状态机不是单独某一个类，而是贯穿数据库、代码、接口的一套规则。
2. 数据库中的 **status** 字段只是状态机的一部分。
3. 真正关键的是“哪些状态能转到哪些状态”。
4. Kafka Consumer、重试、取消、查询接口，都会受到状态机约束。
5. 任务系统一旦变复杂，状态机几乎是必需品。

## 12. 接下来为什么自然衔接到 14 节

理解了状态机之后，下一步非常自然就是理解：

* 为什么任务提交需要幂等
* 为什么要有 **requestId**

因为一旦你开始做：

* 创建任务
* 发 Kafka 消息
* 防止重复提交

你就会立刻遇到：

* 相同请求会不会把任务创建两次
* 同一个任务会不会被执行两次
