# 07 Spring Boot 基础入门

这一模块的目标不是立刻把 Spring Boot 学得很深，而是先让你理解：

- Spring Boot 到底在解决什么问题
- 一个最基础的后端项目通常长什么样
- `Controller`、`Service`、`DTO` 分别负责什么
- 为什么你后面的主项目大概率会以 Spring Boot 为骨架

## 1. 为什么需要 Spring Boot

如果只用最基础的 Java，你当然也能写程序，但当你要做一个真正的后端服务时，会马上遇到很多问题：

- 怎么启动一个 Web 服务
- 怎么处理 HTTP 请求
- 怎么把请求参数转成 Java 对象
- 怎么统一处理异常
- 怎么连接数据库

Spring Boot 的作用可以先简单理解为：

- 帮你快速搭建 Java 后端应用
- 提供很多常用能力的默认配置
- 让你把注意力更多放在业务逻辑本身

## 2. 你可以先怎么理解 Spring Boot

先不要把它想得太复杂。

当前阶段你可以先把 Spring Boot 理解成：

- 一个“帮你快速写后端接口”的框架
- 一个“帮你组织项目结构”的框架
- 一个“帮你把很多重复工作提前做好”的框架

## 3. 一个最基础的 Spring Boot 项目里通常有什么

你后面会经常见到这些角色：

- `Application`：启动类
- `Controller`：接收 HTTP 请求、返回响应
- `Service`：放业务逻辑
- `DTO`：数据传输对象，负责承载请求/响应数据
- `Entity`：数据库中的数据对象
- `Repository` / `Mapper`：和数据库交互

当前阶段先重点记住前 3 个：

- `Controller`：对外
- `Service`：对内
- `Application`：启动入口

## 4. 最常见的分层思路

### `Controller`

负责：

- 接收请求
- 调用 Service
- 返回结果

不建议在这里塞太多复杂业务逻辑。

### `Service`

负责：

- 真正的业务处理
- 组织规则
- 调用数据库层

例如：

- 创建任务
- 校验任务状态
- 统计某个用户的请求次数

## 5. 一个最小接口长什么样

例如你有一个“打招呼”的接口：

```java
@RestController
public class HelloController {

    @GetMapping("/hello")
    public String hello() {
        return "Hello Spring Boot";
    }
}
```

它的含义可以先这样理解：

- `@RestController`：说明这是一个处理 Web 请求的类
- `@GetMapping("/hello")`：当收到 `GET /hello` 请求时，执行这个方法
- 返回值会直接作为响应内容返回

## 6. 为什么要有 Service 层

因为真实项目里，逻辑很快就会变复杂。

如果你把所有逻辑都写在 Controller 里，代码会越来越乱。

例如：

- 参数校验
- 状态判断
- 调数据库
- 记录日志
- 拼装返回结果

这些都堆在一个方法里，会很难维护。

所以更常见的写法是：

- Controller 负责“接请求”
- Service 负责“做事情”

## 7. 一个稍微像样一点的最小结构

```text
com.example.demo
├── DemoApplication.java
├── controller
│   └── TaskController.java
├── service
│   └── TaskService.java
└── dto
    └── TaskResponse.java
```

## 8. 这些内容和你主项目的关系

你后面的主项目里大概率会有这些对应关系：

- `AuthController`：登录、注册
- `TaskController`：提交任务、查任务、取消任务
- `LlmController`：模型调用接口
- `TaskService`：任务逻辑
- `LlmService`：模型调用逻辑

也就是说，你现在学的不是“脱离项目的知识”，而是在提前认识主项目的骨架。

## 9. 当前阶段你先不必死记的内容

下面这些东西后面会学，但现在先知道名字就够：

- 依赖注入
- 配置文件 `application.yml`
- JPA / MyBatis
- 过滤器 / 拦截器
- 全局异常处理
- Bean 生命周期

## 10. 学完这一模块后你应该能做到

- 知道 Spring Boot 是干什么的
- 看懂最基础的 `Controller + Service` 代码结构
- 理解为什么项目要分层
- 能把它和你的主项目规划关联起来

# Spring 相关概念科普

这份文档专门用来回答几个非常常见、也非常容易混淆的问题：

- `Java`、`Spring`、`Spring Boot` 分别是什么
- 什么叫“框架”
- 什么叫“库”
- 什么叫“项目”
- 我们做后端时，到底是在“写框架”还是“写业务项目”

如果这些概念一开始不够清楚，其实很正常。很多初学者第一次接触后端时，都会把这些词混在一起。

## 1. 先给出一句最短答案

- `Java`：编程语言
- `Spring`：Java 生态里非常重要的一套开发框架
- `Spring Boot`：基于 Spring 的快速开发框架/脚手架体系
- 你的项目：基于 Spring Boot 写出来的业务应用

所以你后面真正要做的是：

- 学会使用 Spring Boot
- 用 Spring Boot 搭建自己的后端项目

而不是：

- 从零重写一个 Spring Boot 框架

## 2. 什么是编程语言

编程语言是你写程序时使用的“表达方式”。

例如：

- `Java`
- `Python`
- `Go`
- `C++`

语言本身提供的是基础能力，比如：

- 变量
- 循环
- 类和对象
- 异常处理
- 文件读写

你前面学的 `if`、`for`、类、异常，这些都属于“Java 语言本身”的内容。

## 3. 什么是库（library）

库可以先理解成：

- 别人已经写好的一组可复用代码
- 你需要时，把它拿来调用

例如：

- 一个处理 JSON 的库
- 一个发送 HTTP 请求的库
- 一个加密 JWT 的库

它的特点通常是：

- 你主动调用它
- 你决定什么时候用它
- 它帮你完成某一类具体功能

你可以把“库”理解成“工具箱里的某个工具”。

## 4. 什么是框架（framework）

框架比库更进一步。

你可以先这样理解：

- 库：你调用它
- 框架：它定义整体结构，你把代码放进去，它来调用你

这就是常说的“控制反转”在直觉层面的感觉。

### 一个非常直观的区别

如果你自己写一个普通 Java 程序：

- 你决定程序从哪里开始
- 你决定对象怎么创建
- 你决定每一步怎么串起来

如果你用 Spring Boot 写后端：

- 框架帮你启动 Web 服务
- 框架帮你扫描 `Controller`
- 框架帮你把请求路由到对应方法
- 框架帮你管理很多对象的创建与装配

这时你是在“把代码写进框架规定的结构里”。

## 5. 什么是项目（application / project）

项目就是你真正要交付的应用程序。

例如你的主项目：

- AI 任务中心
- LLM 网关
- JWT 鉴权
- 任务状态管理
- Kafka 异步执行

这些是你的业务目标。

Spring Boot 只是你拿来实现这些目标的开发基础设施。

所以关系是：

- `Java` 是语言
- `Spring Boot` 是框架
- `你的项目` 是用这个框架写出来的应用

## 6. `Java`、`Spring`、`Spring Boot` 的关系

这三个词最容易混。

### `Java`

它是语言。

你写的 `.java` 文件、类、方法、对象，都是 Java。

### `Spring`

它是一个非常大的框架生态。

Spring 解决了很多企业开发中的核心问题，例如：

- 对象管理
- 依赖注入
- 模块解耦
- Web 开发支持
- 数据访问支持

但 Spring 原生体系相对更底层一些，配置和整合成本更高。

### `Spring Boot`

你可以把它理解成：

- Spring 生态里的“快速启动器”
- 帮你把很多默认配置提前做好
- 让你更快搭出可运行的后端服务

所以不是：

- `Spring Boot` 替代了 `Spring`

而更像是：

- `Spring Boot` 建立在 `Spring` 之上
- 它让你更方便地使用 Spring

## 7. 为什么企业里通常不会“从零造一个后端框架”

因为框架层面的东西太重了。

如果你真的从零做一个类似 Spring Boot 的框架，你要处理很多底层问题：

- HTTP 服务启动
- 请求路由
- 对象生命周期管理
- 依赖注入
- 配置管理
- 异常处理机制
- 参数绑定
- 数据库集成
- 日志与扩展机制

这些本身就是大型工程。

对于大多数业务开发岗位来说，真正重要的是：

- 你能否用成熟框架把业务系统做出来
- 你能否写出结构清晰、可维护、可扩展的项目

而不是：

- 你能否自己重写一个 Spring Boot

## 8. 那我们“从零开始做后端”到底是什么意思

这里的“从零开始”，通常不是指：

- 从零发明语言
- 从零发明框架
- 从零发明 HTTP 服务器

而通常是指：

- 从一个空仓库开始
- 初始化一个 Spring Boot 项目
- 自己设计表结构、接口、模块边界
- 一步一步把业务系统搭起来

也就是说，“从零开始做后端项目”和“从零造框架”是两件完全不同的事。

## 9. 你现在这条学习路线里，真正要掌握的是什么

你当前最需要掌握的是：

### 第一层：语言基础

- Java 基础语法
- 面向对象
- 集合
- 异常
- 文件读写

### 第二层：后端基础

- SQL
- HTTP 请求/响应
- REST API
- 登录与鉴权
- 日志
- 配置

### 第三层：框架使用

- Spring Boot 项目结构
- Controller / Service / Repository 分层
- 参数绑定
- 全局异常处理
- 数据库集成

### 第四层：工程能力

- JWT
- Redis
- Kafka
- LLM 接入
- 限流、重试、审计、可观测

这才是你后面主项目真正会落到的能力。

## 10. 一个容易记住的类比

你可以这样类比：

- `Java` 像“语言本身”
- `Spring Boot` 像“成熟的施工脚手架和标准工具”
- `你的项目` 像“你真正要盖的房子”

你现在的任务不是：

- 去重新发明水泥、钢筋、起重机

而是：

- 学会使用这些成熟工具
- 按合理结构把房子盖出来

## 11. 对你当前阶段最重要的结论

请先牢牢记住这几句：

1. `Spring Boot` 是框架，不是你要重写的东西。
2. 你要做的是“基于 Spring Boot 开发项目”。
3. 企业更看重你是否能用成熟框架做出稳定、清晰、可维护的后端系统。
4. 你当前的学习重点应该是“会用框架 + 懂项目结构 + 能实现业务功能”。

## 12. 接下来怎么衔接

有了这层认识之后，你后面再看这些内容会更顺：

- 为什么要有 `Controller`
- 为什么要有 `Service`
- 为什么要接数据库
- 为什么 JWT、Redis、Kafka 都是在“项目层面”接进去的

也就是说，后面我们学的不是“怎么造 Spring Boot”，而是：

- 怎么用 Spring Boot 实现你的 AI 平台后端项目
