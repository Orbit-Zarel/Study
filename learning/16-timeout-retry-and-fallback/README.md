# 16 Timeout、Retry 与 Fallback

这一模块会进入 LLM Gateway 里非常关键的一组“稳定性能力”。

因为当你的项目开始调用外部模型服务时，你很快就会发现：

- 上游不一定永远稳定
- 请求不一定永远能快速返回
- 有时会超时
- 有时会失败

如果没有治理策略，系统会非常脆弱。

所以常见的平台后端会考虑三件事：

- timeout
- retry
- fallback

## 1. 什么是 timeout

timeout 可以先理解成：

- 一次请求最多允许等待多久

如果超过这个时间还没拿到结果，就不要一直傻等下去。

例如：

- 你设置超时时间为 30 秒
- 如果 30 秒还没返回，就认为请求超时

## 2. 为什么一定要有 timeout

因为如果没有超时控制，可能会出现：

- 某个上游请求一直卡住
- 线程长期被占用
- 用户一直等不到响应
- 服务整体吞吐被拖垮

所以 timeout 本质上是在保护：

- 用户体验
- 服务资源
- 系统稳定性

## 3. 什么是 retry

retry 可以先理解成：

- 第一次请求失败后，再试一次或几次

例如：

- 第一次请求因为网络抖动失败
- 第二次可能就成功了

## 4. 为什么 retry 有价值

因为有些错误不是“永久错误”，而是“临时错误”。

例如：

- 短暂网络波动
- 上游偶发超时
- 某些 5xx 服务异常

这类错误有时再试一次就好了。

## 5. 为什么不能无脑重试

这是非常重要的工程意识。

并不是所有失败都应该重试。

例如：

- 参数本来就错了
- API Key 无效
- 请求格式不对

这种错误你重试 10 次也没用。

所以要先区分：

- 可重试错误
- 不可重试错误

## 6. 一个很基础的重试判断直觉

你当前阶段可以先粗略记成：

### 常见可重试

- timeout
- 网络异常
- 部分 5xx

### 常见不可重试

- 请求参数错误
- 鉴权失败
- 供应商返回明确 4xx 业务错误

## 7. 什么是 fallback

fallback 可以先理解成：

- 主方案失败后，切到备用方案

在你的项目里，最常见的场景就是：

- OpenAI 调用失败
- 切换到通义千问

也就是说：

- retry 是“再试一次同样的方案”
- fallback 是“换一个方案再试”

## 8. retry 和 fallback 的区别

很多人一开始容易把它们混在一起。

你可以先这样记：

### retry

- 还是同一个 provider
- 只是再尝试一次

### fallback

- 换一个 provider 或备用路径

例如：

- retry：OpenAI 超时，再试一次 OpenAI
- fallback：OpenAI 失败后改走 Qwen

## 9. 一个最小治理流程

你可以先把它想成：

1. 收到模型调用请求
2. 设置超时时间
3. 调用主 provider
4. 如果是可重试错误，就有限次数重试
5. 如果重试后仍失败，并且启用了 fallback
6. 再尝试备用 provider
7. 把最终结果和治理过程记录下来

## 10. 为什么 fallback 很有价值

因为它能提升系统韧性。

例如：

- OpenAI 短时波动
- 但 Qwen 还能正常工作

这时如果允许 fallback，系统就不至于直接失败。

## 11. 为什么 fallback 也不能乱用

因为切换 provider 也会带来问题：

- 模型效果可能不同
- 响应格式可能差异更大
- 成本可能变化
- 某些任务可能不适合自动切换

所以实际项目里通常要明确：

- 哪些场景允许 fallback
- fallback 发生时是否要记录

## 12. 为什么这三者是 LLM Gateway 的核心能力

因为 LLM Gateway 不只是“把请求发出去”。

它真正的价值之一就在于：

- 对上游模型调用做稳定性治理

也就是：

- 不让调用方自己分散处理超时
- 不让每个业务服务自己各写一套重试逻辑
- 统一管理 fallback 行为

## 13. 当前阶段最值得记住的几句话

请先记住：

1. timeout 是为了防止一直卡住。
2. retry 是为了应对临时性失败。
3. fallback 是为了在主路径不可用时换备用路径。
4. 不是所有错误都应该重试。
5. fallback 发生时通常应该被记录下来。

## 14. 学完这一模块后你应该能做到

- 理解 timeout、retry、fallback 各自是做什么的
- 理解 retry 和 fallback 的区别
- 理解为什么 LLM Gateway 需要统一做这些治理
- 为后续学习限流、审计、可观测打基础
