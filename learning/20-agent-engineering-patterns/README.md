# 20 Agent 工程模式

这一模块开始正式进入 Agent 开发本身。

你前面已经学过：

- `FastAPI`
- `MCP`
- 流式输出
- Hook
- Session
- 多 Agent 原型
- 限流、熔断、日志、线程池

这一节要做的，是把这些零散能力整理成更稳定的 Agent 工程模式。

## 1. 什么是 Agent 工程

当前阶段先用最实用的角度理解：

- 不是“让模型回答问题”这么简单
- 而是围绕模型构建一套可执行、可约束、可恢复、可观测的任务系统

也就是：

- 什么时候直接回答
- 什么时候调工具
- 工具失败怎么办
- 上下文太大怎么办
- 多个子任务怎么拆
- 最终结果怎么汇总

这些都属于 Agent 工程问题。

## 2. 为什么要学习“模式”

因为 Agent 项目很容易一开始能跑，后面越来越乱。

常见问题：

- prompt 越写越长
- 工具越来越多，模型乱调
- 返回结果太大，导致上下文膨胀
- 某一步失败后，整条链路不知道怎么恢复
- Web 接口、CLI、飞书机器人各写一套逻辑

所以你需要的不是“再多写几个 demo”，而是先形成稳定模式。

## 3. 一个最小 Agent 链路里通常有什么

你可以先记成这几个部分：

1. 用户输入
2. Agent 判断是否需要工具
3. 调用工具
4. 解析工具结果
5. 决定继续调工具还是直接回答
6. 输出最终结果

如果系统再复杂一点，就会继续增加：

- 会话状态
- 安全限制
- 日志与 traceId
- 错误恢复
- 多 Agent 协作

## 4. 单 Agent 模式

这是最基础的一种模式。

特点：

- 只有一个 Agent
- 一个 Agent 同时负责理解问题、决定工具、整理答案

优点：

- 实现简单
- 成本低
- 链路短

缺点：

- 上下文容易膨胀
- 工具太多时不稳定
- 复杂任务容易混乱

当前很多内部工具型项目，第一版都适合先从单 Agent 开始。

## 5. Planner-Executor 模式

这是非常常见的第二层模式。

做法是把角色拆开：

- `Planner` 负责拆任务
- `Executor` 负责执行子任务

例如：

```text
用户问题
  -> Planner 输出任务列表
  -> Executor 逐个执行
  -> 汇总结果
```

这样做的价值是：

- 拆分复杂问题
- 降低单次上下文复杂度
- 让每个执行器只关注一类事情

## 6. Planner-Executor-Synthesizer 模式

这是你后面最值得掌握的一种 Agent 工程骨架。

角色分工：

- `Planner`：负责规划
- `Executor`：负责查数据、调工具、跑任务
- `Synthesizer`：负责根据中间结果生成最终回答

这个模式适合：

- 数据查询型 Agent
- 报告生成型 Agent
- 多数据源聚合型 Agent

## 7. 为什么中间结果不要全塞回上下文

这是 Agent 工程里非常关键的点。

很多系统出问题，不是模型不聪明，而是中间结果太大。

例如：

- 一次查回 20 条长文本
- 再把全部结果原样塞回模型上下文
- 模型继续规划、继续调工具

很快就会出现：

- 上下文变大
- 成本变高
- 速度变慢
- 结果不稳定

所以一个常见模式是：

- 中间结果先落到文件、缓存、数据库
- 后续 Agent 只读取必要摘要

## 8. “时间换空间”模式

这是你做多 Agent 时很常见的工程思路。

做法是：

- 不让一个 Agent 一次处理所有结果
- 而是拆成多个范围更小的执行器

例如：

- PG Agent 只查一个 schema
- Mongo Agent 只查一个 collection
- 文档 Agent 只处理某一批文件

这样做会增加步骤数，但能控制上下文体积。

## 9. Tool Guard 模式

Agent 能调工具，不代表应该毫无限制。

所以通常需要一层工具保护。

常见做法：

- 黑名单工具过滤
- SQL 关键词检查
- 参数校验
- 最大工具调用次数限制

它解决的是：

- 危险调用
- 死循环调用
- 无意义重复调用

## 10. Structured Output 模式

一旦进入多 Agent 或多步骤系统，结构化输出就非常重要。

因为下游代码需要可靠解析上游结果。

例如规划器输出：

```json
{
  "tasks": [
    {"executor": "postgres", "goal": "查询 agent_prompt"},
    {"executor": "mongodb", "goal": "查询日志集合"}
  ]
}
```

这类输出比自由文本更适合工程系统。

## 11. Fallback 模式

Agent 项目里的 fallback 不只是模型供应商切换。

还可能包括：

- 规划失败时的关键词兜底分配
- JSON 解析失败时的默认结构
- 工具返回空结果时的二次查询
- 主路径失败时退回简单回答

所以 fallback 要理解成：

- 主策略失败时的可控降级路径

## 12. Human-in-the-Loop 模式

并不是所有 Agent 都应该全自动。

很多真实场景更适合：

- 自动生成草稿
- 人工审核
- 再确认落库或发送

这在这些场景尤其常见：

- 金融分析
- 报告生成
- 医疗场景
- 风险控制

## 13. Session + Memory 模式

如果 Agent 有连续对话，就要考虑：

- 会话怎么保存
- 历史消息保留多少
- 什么时候摘要历史
- 什么时候清理上下文

否则长期对话会越来越重。

所以会话管理本身就是 Agent 工程的一部分。

## 14. 可观测模式

Agent 项目排障时，至少要能回答这些问题：

- 本轮到底调了哪些工具
- 哪一步失败
- 是模型没调工具，还是工具报错
- 上下文是不是爆了
- 最终回答是哪个模型生成的

所以至少要记录：

- `traceId`
- `sessionId`
- `toolName`
- `latencyMs`
- `model`
- `status`

## 15. 当前阶段你要先掌握的 5 个模式

先抓住这 5 个：

1. 单 Agent 模式
2. Planner-Executor 模式
3. Planner-Executor-Synthesizer 模式
4. Tool Guard 模式
5. Structured Output 模式

这 5 个已经足够支撑你继续学 `LangGraph` 和 `RAG`。

## 16. 学完这一模块后你应该能做到

- 能区分单 Agent 和多 Agent 的适用场景
- 理解为什么要拆 Planner / Executor / Synthesizer
- 理解上下文膨胀为什么是 Agent 工程的核心问题
- 理解为什么中间结果常常需要落盘或缓存
- 能用“模式”而不是“堆 prompt”的方式看 Agent 系统
