# 08 JWT 与鉴权基础

这一模块的目标是先帮你建立最基础的安全与身份概念，而不是一上来就记很多细节。

你后面的主项目里，一定会遇到这些问题：

- 用户怎么登录
- 服务端怎么知道“你是谁”
- 为什么有的接口登录后才能访问
- JWT 到底是什么

## 1. 先区分两个很容易混淆的词

### 认证（Authentication）

认证解决的是：

- “你是谁”

例如：

- 输入用户名密码登录
- 服务端验证账号密码是否正确

### 授权（Authorization）

授权解决的是：

- “你能做什么”

例如：

- 普通用户只能看自己的任务
- 管理员可以看所有人的任务

可以先记一句：

- 认证：确认身份
- 授权：决定权限

## 2. 为什么后端需要登录机制

因为后端不能默认信任任何请求。

如果没有登录和鉴权，任何人都可以：

- 查看别人数据
- 提交恶意请求
- 调用收费接口
- 冒充其他用户

所以系统通常要有“受保护接口”：

- 登录前不能访问
- 登录后拿到身份凭证才能访问

## 3. 什么是 Token

Token 可以先理解成：

- 登录成功后，服务端发给客户端的一张“临时通行证”

之后客户端每次请求时都带上它，服务端据此识别身份。

## 4. 什么是 JWT

JWT 全称是 `JSON Web Token`。

当前阶段你可以把它简单理解成：

- 一种常见的 Token 格式
- 它里面可以携带用户信息和过期时间
- 服务端可以验证它是否被篡改

你现在先不用把它想得太神秘。

## 5. JWT 一般长什么样

一个 JWT 通常像这样：

```text
xxxxx.yyyyy.zzzzz
```

它通常由 3 部分组成：

- Header
- Payload
- Signature

### Header

描述 Token 的基本信息，例如：

- 类型是不是 JWT
- 使用什么签名算法

### Payload

里面通常放一些业务相关信息，例如：

- 用户 ID
- 用户名
- 角色
- 过期时间

### Signature

签名部分用来防止别人随便篡改 Token 内容。

## 6. JWT 的基本工作流程

你可以先把整个过程理解为：

1. 用户用用户名和密码请求登录
2. 服务端验证通过
3. 服务端生成 JWT 并返回给客户端
4. 客户端之后访问接口时，在请求头里带上 JWT
5. 服务端验证 JWT 是否有效
6. 如果有效，就允许访问受保护接口

常见请求头格式：

```text
Authorization: Bearer <token>
```

## 7. 什么是受保护接口

受保护接口就是：

- 不能匿名访问
- 需要带合法身份凭证才能访问

例如：

- `GET /api/tasks`
- `POST /api/llm/generate`

而这些接口通常不受保护：

- `POST /api/auth/login`
- `POST /api/auth/register`

## 8. JWT 和 Session 的区别

这是一个非常经典的问题。

### Session 思路

可以先简单理解为：

- 登录后，服务端保存一份会话状态
- 客户端只拿一个会话标识
- 之后服务端根据这个标识去查会话

### JWT 思路

可以先简单理解为：

- 登录后，服务端把一些身份信息编码进 Token
- 客户端带着 Token 来访问
- 服务端重点验证这个 Token 是否合法

### 初学阶段怎么记

可以先粗略记成：

- Session：状态更多保存在服务端
- JWT：身份信息更多体现在客户端携带的 Token 里

## 9. JWT 适合你后面项目的原因

你当前规划里的主项目比较偏“平台后端 / API 服务”，JWT 很常见，原因包括：

- 和前后端分离项目搭配自然
- 适合 REST API 风格
- 请求头携带方式比较统一
- 做内部平台接口时很常见

## 10. 当前阶段需要重点理解的风险意识

学 JWT 时，不要只记“怎么发 token”，还要有一些安全意识：

- Token 不能泄露
- 要设置过期时间
- 敏感信息不要直接明文塞进 Payload
- 服务端要校验签名，不能只看内容

## 11. 学完这一模块后你应该能做到

- 区分认证和授权
- 理解 JWT 是做什么的
- 看懂最基础的登录 -> 发 Token -> 带 Token 访问接口的流程
- 知道为什么后端要有受保护接口
