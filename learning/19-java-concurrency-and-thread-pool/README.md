# 19 Java 并发与线程池基础

这一模块开始补你后面做后端和异步任务时绕不开的一块：

- 并发
- 线程池

前面你已经理解了 Kafka、状态机、幂等、LLM Gateway、timeout、retry、限流、熔断。

但这些能力最终都需要代码真正跑在某些执行线程里。

所以这一节的目标是先把最基础的执行模型补起来。

## 1. 什么是线程

当前阶段先把线程理解成：

- 程序里的一个执行单元

一个后端服务并不是同一时刻只能做一件事。

例如：

- 用户 A 发请求
- 用户 B 也发请求
- 后台还有任务在消费消息

这些事情通常都依赖多个线程并发执行。

## 2. 什么是并发

并发先用最实用的角度理解：

- 系统在同一时间段内处理多件事情

对后端来说，这几乎是基本盘。

因为你的服务不可能只服务一个请求。

## 3. 为什么不能一直手动 new Thread

理论上你可以每来一个任务就 `new Thread()`。

但工程上通常不这么做，因为：

- 线程创建有成本
- 数量失控会拖垮系统
- 不方便统一管理

所以实际项目里更常见的是：

- 线程池

## 4. 什么是线程池

线程池可以先理解成：

- 预先准备一批可复用线程
- 任务来了就交给它们执行

这样做的价值是：

- 复用线程
- 控制并发数量
- 统一任务调度

## 5. 线程池和你的主项目有什么关系

关系非常直接。

例如后面这些场景都跟它有关：

- HTTP 请求处理
- 异步任务执行
- 超时控制
- 后台重试任务
- 批处理任务调度

即使你后面用了 Spring Boot、Kafka、消息队列，底层仍然绕不开线程执行模型。

## 6. 一个最小线程池直觉

你先记住这条链路：

1. 创建线程池
2. 提交任务
3. 空闲线程取任务执行
4. 执行结束后线程回到池里等待下一个任务

## 7. 什么是异步提交

异步提交可以先理解成：

- 调用方把任务交出去后，不一定等待它当场执行完

例如：

- 接口先返回“任务已提交”
- 真正执行在后台线程继续处理

这和你前面 Kafka 那一节的思路是同方向的。

## 8. 并发带来的新问题是什么

并发不是免费能力。

你后面会逐渐遇到这些问题：

- 共享数据是否安全
- 状态更新是否冲突
- 重复执行如何避免
- 线程池是否被打满

所以并发通常会和这些概念一起出现：

- 幂等
- 锁
- 原子性
- 任务队列
- 拒绝策略

当前阶段先建立直觉，不展开太深。

## 9. 当前阶段你应该先理解哪些点

先抓住这几个：

1. 线程是执行单元
2. 并发是同时段处理多件事
3. 线程池比手动创建线程更适合后端服务
4. 异步任务最终还是要落到线程执行

## 10. 学完这一模块后你应该能做到

- 理解线程池为什么是后端基础设施的一部分
- 看懂最小的任务提交与执行流程
- 理解并发为什么会引出状态一致性和幂等等问题
- 能把线程池和 Kafka/异步任务/超时控制连起来看
