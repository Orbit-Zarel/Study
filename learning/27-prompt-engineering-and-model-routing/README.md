# 27 Prompt Engineering 与 Model Routing

这一模块开始补 Agent 系统里另一条很关键的主线：

- prompt 设计
- model routing

前面你已经补了：

- LangChain / LangGraph
- RAG
- 检索优化
- MCP / A2A

这一节要补的是：

- 为什么 prompt 不是随便写一句话
- 为什么不同任务不一定用同一个模型
- Agent 系统里如何做模型选择与降级

## 1. 什么是 Prompt Engineering

当前阶段先把它理解成：

- 为了让模型更稳定、更准确、更可控地完成任务，对输入指令做结构化设计

重点不是“写得花”，而是：

- 任务边界清楚
- 输出要求明确
- 限制条件明确
- 失败行为可控

## 2. 为什么 prompt 很重要

很多 Agent 系统的问题，不是工具坏了，也不是模型不够强，而是 prompt 太含糊。

常见问题：

- 任务目标不清楚
- 输出格式不清楚
- 调工具时机不清楚
- 没证据时该怎么回答不清楚

结果就是：

- 模型乱答
- 模型不调工具
- 模型输出格式不稳定

## 3. prompt 通常在约束什么

一个更完整的 prompt，通常会约束这些内容：

1. 角色
2. 任务目标
3. 输入背景
4. 输出格式
5. 工具使用规则
6. 错误与无结果时的处理方式

## 4. 什么是系统提示词

系统提示词可以先理解成：

- 整个 Agent 的高层行为规则

例如：

- 你是谁
- 你擅长什么
- 你不能做什么
- 你什么时候要调工具

它更像长期规则，不是一次性用户问题。

## 5. 什么是任务提示词

任务提示词更偏当前这次请求本身。

例如：

- 请分析某家公司本季度利润变化
- 请只基于给定文档回答
- 请输出 JSON

所以可以粗略理解成：

- system prompt 管长期行为
- task prompt 管本次任务

## 6. 什么是输出约束

如果一个系统要把模型输出继续交给代码处理，就必须做输出约束。

例如要求：

- 输出 JSON
- 输出固定字段
- 输出分点总结
- 输出引用来源

这和你前面学过的 structured output 是同一条主线。

## 7. 什么是工具调用 prompt

当 Agent 需要调工具时，prompt 往往还要额外说明：

- 什么时候应该调用工具
- 什么时候不要调用工具
- 参数该怎么理解
- 没查到结果时怎么处理

这部分如果写得差，常见后果是：

- 模型只说“我来查一下”，但不真正调工具
- 模型参数乱填
- 模型明明没证据还直接编答案

## 8. 什么是 model routing

model routing 可以先理解成：

- 根据任务类型、成本、速度、质量要求，决定这次该用哪个模型

也就是：

- 不是所有请求都固定走一个模型

## 9. 为什么需要 model routing

因为不同模型通常在这些维度上不一样：

- 成本
- 延迟
- 上下文长度
- 结构化输出稳定性
- 工具调用能力
- 推理能力

如果所有任务都走同一个最强模型，往往会带来：

- 成本高
- 速度慢
- 资源浪费

## 10. 常见的 routing 维度

当前阶段先抓住这些：

### 按任务类型

例如：

- 简单问答 -> 小模型
- 复杂分析 -> 大模型
- 结构化抽取 -> 更稳定的结构化模型

### 按工具需求

例如：

- 需要强 tool use 的任务 -> 工具能力更强的模型

### 按上下文长度

例如：

- 长文档总结 -> 上下文更大的模型

### 按成本与延迟

例如：

- 高频低价值请求 -> 便宜快速模型

## 11. 什么是 fallback routing

fallback routing 可以先理解成：

- 主模型失败后，切到备用模型

例如：

- 主模型超时
- 主模型 5xx
- 主模型工具调用失效

这时系统不一定直接失败，而是：

- 按规则切到备用模型

## 12. prompt 和 routing 的关系

这两者其实是联动的。

因为：

- 不同模型对 prompt 的敏感度不同
- 不同模型对工具调用协议的遵循度不同
- 不同模型对结构化输出的稳定性不同

所以真实系统里常常不是：

- 一个 prompt 适配所有模型

而是：

- prompt 模板和 model routing 策略一起设计

## 13. 一个最小 routing 直觉

你可以先记成：

```text
用户请求
  -> 判断任务类型
  -> 选择模型
  -> 选择 prompt 模板
  -> 执行
  -> 失败时 fallback
```

## 14. 什么是 prompt 模板分层

很多系统不会只维护一份 prompt。

更常见的是分层：

- system prompt
- task prompt
- tool prompt
- output format prompt

这样做的好处是：

- 复用
- 易维护
- 易针对不同场景微调

## 15. 当前阶段最该先掌握什么

先抓住这几个：

1. prompt 设计的目标是稳定和可控
2. system prompt 和 task prompt 作用不同
3. 工具调用 prompt 要明确规则
4. 不是所有任务都该走同一个模型
5. routing 和 fallback 都属于平台治理能力

## 16. 学完这一模块后你应该能做到

- 理解 prompt engineering 在 Agent 系统里的位置
- 能区分 system prompt、task prompt、tool prompt、output prompt
- 理解为什么需要 model routing
- 理解 fallback model 的作用
- 能把 prompt 和 routing 放到平台级视角里理解
