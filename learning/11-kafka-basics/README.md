# 11 Kafka 基础入门

这一模块的目标是帮助你建立“异步任务”这条主线的基础直觉。

你后面的主项目里，很重要的一部分就是：

- 提交任务
- 投递消息
- 异步处理
- 查询状态

而 Kafka 就是这个流程里非常常见的基础设施之一。

## 1. 为什么有时不能同步直接做完

先看一个直观场景：

- 用户提交一个任务
- 这个任务要调用大模型
- 大模型调用可能比较慢
- 甚至还有失败重试、结果存储这些后续操作

如果你把这些事情全都放在一次 HTTP 请求里同步完成，可能会出现：

- 用户等待太久
- 接口容易超时
- 系统吞吐能力变差
- 某些失败重试不方便处理

所以就会出现一种常见思路：

- 先把任务接下来
- 再异步处理

## 2. 什么是消息队列

消息队列可以先理解成：

- 一个“消息中转站”
- 生产者把消息放进去
- 消费者再从里面取出来处理

你可以把它想成：

- 前面的人负责“提交任务单”
- 后面的人负责“按任务单执行”

## 3. Kafka 是什么

Kafka 可以先简单理解成：

- 一个高吞吐的分布式消息系统
- 常用于异步任务、日志流、事件流等场景

你当前阶段先抓住这个直觉就够：

- Kafka 常被用来把“请求接收”和“任务执行”拆开

## 4. 什么是 Producer 和 Consumer

### Producer

Producer 就是：

- 负责发送消息的人或程序

例如：

- 你的后端接口收到“创建任务”请求后
- 把 `taskId` 发到 Kafka

### Consumer

Consumer 就是：

- 负责消费消息并处理业务的人或程序

例如：

- 从 Kafka 里取出 `taskId`
- 查询任务
- 执行 LLM 调用
- 更新任务状态

## 5. 什么是 Topic

Topic 可以先理解成：

- Kafka 中某一类消息的“主题”或“队列名字”

例如：

- `task-execute`
- `llm-request`
- `user-events`

如果你把 Kafka 想成一个大型消息系统，那么 Topic 就像里面的不同通道。

## 6. 一个最基础的异步任务流程

你可以先把它记成下面这条链路：

1. 用户调用 `POST /api/tasks`
2. 服务端创建任务记录，得到 `taskId`
3. Producer 把 `taskId` 发送到 Kafka
4. Consumer 从 Kafka 获取这条消息
5. Consumer 执行具体业务
6. Consumer 更新任务状态为成功或失败
7. 用户之后可以通过接口查询任务状态

这条链路和你主项目规划是直接对应的。

## 7. 为什么 Kafka 对你的项目有价值

因为你的项目后面会有很多“适合异步做”的事情：

- 调用耗时的 LLM 请求
- 做批处理任务
- 做失败重试
- 做状态更新

Kafka 帮你做到的是：

- 接口先快速返回
- 真正的耗时逻辑交给后台慢慢处理

## 8. 当前阶段先认识几个关键词

### 异步

表示：

- 提交之后不一定马上拿到最终结果
- 真正执行在后面进行

### 解耦

表示：

- 提交请求的模块
- 和执行任务的模块

不必强绑定在一起。

### 削峰

表示：

- 突然很多请求进来时
- 先放进消息队列缓冲
- 后面的消费者按自己的节奏处理

## 9. 为什么消息队列也会有新问题

引入 Kafka 不是没有代价的。

你后面会逐渐遇到这些问题：

- 消息会不会重复消费
- 消费失败怎么办
- 状态怎么更新
- 用户怎么知道任务是否完成

所以 Kafka 往往会和这些概念一起出现：

- 幂等
- 重试
- 状态机
- 死信队列

当前阶段先不用展开太深，只要先知道：

- Kafka 带来了异步能力
- 也带来了额外的工程复杂度

## 10. 幂等为什么重要

这是消息队列场景里非常重要的工程意识。

如果同一条消息被重复消费，而你的代码没有做好保护，可能会导致：

- 重复调用大模型
- 重复扣费
- 重复写结果

所以后面做 Consumer 时，经常要考虑：

- 同一任务是否已经处理过
- 相同消息再次到来时是否安全

## 11. 学完这一模块后你应该能做到

- 理解 Kafka 是做什么的
- 看懂 Producer / Consumer / Topic 这些最基础概念
- 理解为什么异步任务对平台后端很重要
- 知道 Kafka 往往会和幂等、重试、状态更新一起出现
