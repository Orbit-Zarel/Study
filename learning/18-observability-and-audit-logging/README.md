# 18 可观测性与审计日志

这一模块开始补平台后端里非常容易被忽略，但实际排障时最关键的一部分：

- 可观测性
- 审计日志

前面你已经学了接口、鉴权、异步任务、状态机、幂等、LLM Gateway、timeout、retry、fallback、限流、熔断。

这些能力如果没有日志和链路信息，出了问题会很难排查。

## 1. 什么是可观测性

当前阶段先用最短定义理解：

- 当系统出问题时，你能从日志和指标里快速知道发生了什么

对后端项目来说，最先落地的一层通常不是很复杂的监控平台，而是：

- 结构化日志
- traceId
- 关键字段统一记录

## 2. 为什么普通打印日志不够

例如你只写：

- 调用成功
- 调用失败

这种日志几乎没有排障价值。

因为你不知道：

- 哪个用户的请求
- 哪个接口
- 哪个 provider
- 用了哪个 model
- 耗时多少
- 是超时、限流还是鉴权失败

所以平台后端的日志通常要能回答这些问题。

## 3. 什么是结构化日志

结构化日志可以先理解成：

- 日志不是随便拼一句话
- 而是按固定字段输出

例如：

```text
traceId=trace-1001 userId=user-7 provider=openai model=gpt-4o-mini latencyMs=812 status=SUCCESS
```

这样做的好处是：

- 人能看
- 程序也能搜
- 后面接日志平台更方便

## 4. 什么是 traceId

traceId 可以先理解成：

- 一次请求链路的唯一编号

例如一个请求经过：

- API 网关
- 业务服务
- LLM Gateway
- 外部 provider

这几层日志如果都带同一个 `traceId`，排查时就能把它们串起来。

## 5. 什么是审计日志

审计日志偏向记录：

- 谁在什么时间做了什么操作

这和普通技术日志有重叠，但关注点不完全一样。

例如在你的主项目里，审计日志会很关心：

- 谁调用了 LLM 接口
- 调用了哪个 provider 和 model
- 是否发生 fallback
- token 用量是多少
- 请求最终成功还是失败

## 6. 为什么 LLM Gateway 尤其需要审计

因为它往往涉及：

- 费用
- 配额
- 故障追踪
- 供应商路由
- 安全审查

如果后面有人问：

- 为什么这个用户请求很慢
- 为什么今天 OpenAI 用量激增
- 为什么某个接口频繁 fallback 到 Qwen

这些都要靠审计和日志来回答。

## 7. 当前阶段日志至少该记什么

先记住最小集合：

- `traceId`
- `userId`
- `requestId`
- `provider`
- `model`
- `latencyMs`
- `status`
- `errorCode`

这和你的主计划文档已经对齐。

## 8. 什么信息不该直接打进日志

这部分同样重要。

例如：

- 原始 API Key
- 完整敏感 prompt
- 用户隐私数据

日志不是越多越好，而是：

- 关键字段要够用
- 敏感信息要克制

## 9. 它和前面的模块怎么连起来

你前面学的很多能力，最终都要靠可观测性落地：

- JWT 失败要有错误码
- Kafka 消费失败要能追踪任务
- 幂等命中要能查到 requestId
- LLM fallback 要能记录 provider 切换
- 限流和熔断要能看出是谁被拒绝、为什么被拒绝

## 10. 当前阶段你应该先理解哪些点

先抓住这几个：

1. 可观测性首先是排障能力
2. 结构化日志优先于随意打印
3. traceId 用来串联一条请求链路
4. 审计日志更关心操作记录、用量、责任归属

## 11. 学完这一模块后你应该能做到

- 理解为什么平台后端必须统一日志字段
- 看懂 traceId 在一条链路中的价值
- 能区分技术日志和审计日志的关注点
- 能把关键日志字段带进你后面的主项目设计
