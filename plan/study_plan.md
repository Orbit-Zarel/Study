## 学习计划（面向岗位的可执行版）

### 0. 目标与约束

- **目标岗位画像（偏后端/AI平台/Agent工程）**：
  - JD 高频关键词：`Spring Boot`、`微服务`、`Redis`、`MQ(Kafka/RabbitMQ)`、`并发/线程池`、`JVM`、`Linux`、`SQL`
  - 加分项：`Go`、`C++(性能/并发/系统方向)`、`混合开发(pybind11)`、`CUDA/HPC(长期)`
- **学习方式**：边实习边学
- **时间预算**：每天 `1-2h`（工作日），周末可补 `2-4h`
- **原则**：每个模块都必须沉淀到“能写进简历”的产出

#### 0.0 技术选型（已确认）

- **任务队列**：Kafka
- **优先落地能力点**：LLM 适配层（先不做 RAG）
- **LLM 供应商（已确认）**：OpenAI（国外）+ 通义千问（国内）
- **鉴权方式（已确认）**：JWT
- **学习节奏**：先覆盖经典场景（可复用的 POC），再把 POC 逐步合并成一个主项目

#### 0.2 LLM 适配层（OpenAI + 通义千问）的设计要点（建议你先想清楚）

- **统一抽象**：
  - 输入：`model`、`messages`、`temperature`、`maxTokens`、`responseFormat`（如 json）、`timeoutMs`
  - 输出：`text`/`json`、`usage`（tokens）、`provider`、`model`、`latencyMs`、`requestId`
- **供应商路由策略**（先最简单）：
  - 默认按 `provider` 显式指定（`openai` / `qwen`）
  - 可选 fallback：OpenAI 失败 -> 通义（或反之），但要记录清楚“是否发生降级”
- **可靠性**：
  - 超时：服务端必须可配置（建议默认 30s，按场景调整）
  - 重试：只对“可重试错误”重试（如网络超时/5xx），并限制次数（如 2 次）
  - 限流：
    - 对外（用户请求）限流：保护你的服务
    - 对内（调用供应商）限流：保护 Key 与供应商配额
- **安全与合规**：
  - Key 不写入代码仓库；从环境变量/配置中心读取
  - 日志不打印原始 API Key 与敏感内容；必要时对 prompt 做脱敏
- **可观测性（你要能排障）**：
  - 日志字段建议：`traceId`、`userId`、`taskId`、`provider`、`model`、`latencyMs`、`status`、`errorCode`

#### 3.1 主项目（当你完成 3-5 个场景后再合并）

- **主项目名称（建议）**：AI 任务中心 + LLM 网关（Spring Boot）
- **项目使命（它用来做什么）**：

  - 在网关能力稳定后，再把耗时/不稳定的调用扩展为“可追踪的异步任务”（Kafka），用于批处理与平台化任务。
- **目标用户（先从你自己开始）**：
  - 你自己的 AI 应用/Agent（后续任何项目都可以调用这个网关）
  - 模拟企业内部调用方（不同用户/不同配额/不同模型策略）

  - 作为平台方，我希望限制某个用户的调用频率与配额，并可追溯每次调用成本与失败原因。
  - 作为调用方，我希望提交一个异步任务后能查询状态、拿到结果，并在失败后可重试。
- **合并策略（POC -> 主项目）**：
  - 先把场景1-5 合并成“任务中心 + Kafka 执行器 + 状态机”（平台骨架）

#### 3.4 异步任务与状态机（你要能讲清楚）

- **任务状态（示例）**：`PENDING` -> `QUEUED` -> `RUNNING` -> `SUCCEEDED/FAILED/CANCELED`
- **核心链路（必须实现最小闭环）**：
  - `POST /api/tasks` 创建任务（落库 `tasks`，返回 `taskId`）
  - Producer 投递 Kafka 消息（包含 `taskId`）
  - Consumer 拉取消息：
    - 先做幂等判断（避免重复消费导致重复执行）
    - 更新任务为 `RUNNING`
    - 执行业务（例如：调用 LLM / 跑一个 ETL 处理）
    - 成功写入 `task_results` 并置 `SUCCEEDED`；失败记录原因并置 `FAILED`
- **失败策略（先做最简单）**：
  - 可人工重试（`/retry`）
  - 可记录 `retry_count`（后续再扩展自动重试/死信队列）
